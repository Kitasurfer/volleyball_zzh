# volleyball_zzh Project Rules

## Scope
Эти правила применяются ко всему репозиторию `volleyball_zzh`, включая frontend (Vite + React + Tailwind), голосового ассистента, Supabase/документацию и вспомогательные сервисы.

## Общие практики
- Всегда изучайте существующие AGENTS/workflows перед началом задачи. Если документация не готова, сначала запускайте соответствующий workflow (например, `/voice_architect`).
- Используйте `pnpm` (заявленный package manager) для всех npm-скриптов. Не смешивайте с npm/yarn.
- Изменения должны быть атомарными: одна функциональность/фикс на PR.
- Всегда сохраняйте типобезопасность (TypeScript strict). Любой `any` требует явного обоснования/комментария.
- Любой новый пользовательский текст должен быть локализован (см. раздел «Мультиязычность»).
- Не удаляйте и не перезаписывайте чужие изменения без согласования. Если увидели неожиданные правки — останавливаемся и уточняем у пользователя.

## Мультиязычность (DE/EN/RU/IT)
- Все пользовательские тексты живут в `src/data/translations.ts` или в локальных `content` объектах (например, страниц). При добавлении новой строки необходимо:
  1. Добавить/обновить значение для **каждого** языка (`de`, `en`, `ru`, `it`).
  2. Обновить типы/интерфейсы, если структура изменилась.
  3. Проверить, что компонент берёт текст через `useLanguage()`/`t`.
- Никогда не жёстко кодируйте строки в JSX без привязки к языку, если их видит пользователь.
- Любые ошибки/баннеры/ARIA-лейблы также должны иметь перевод.

## UI/UX и стили
- Tailwind — основной способ стилизации. Соблюдайте существующую дизайн-систему (`src/components/ui`).
- Для фоновых изображений и hero-блоков используйте прозрачные оверлеи и drop-shadow вместо повышения яркости исходников.
- Визуальные изменения проверяйте в desktop и mobile (минимум 375px ширина).
- Сохраняйте указанные в инструкциях цвета/фирменные элементы. Новые цвета/градиенты согласовывайте через дизайнера или пользователя.

## Медиа и SEO
- Любое новое изображение должно лежать в `public/images/` и иметь запись в `public/images/image_meta.json` (источник, caption, размеры).
- Если картинка используется на страницах/Seo-компоненте, указывайте путь вида `/images/<file>` и обновляйте `image_meta.json`.
- Проверяйте, чтобы `Seo` получал правильный `title`, `description`, `imagePath` и JSON-LD.

## Голосовой ассистент
- Для каталога `src/components/chatbot/voice/` действуют инструкции `AGENTS.md`. Ключевые пункты:
  - Web Speech API для STT, локальный Chatterbox Service для TTS.
  - Разделяйте UI и логику (hooks в `/hooks`, компоненты в `/ui`).
  - Полная очистка MediaRecorder/AudioContext, отсутствие утечек.
  - Ошибки/состояния должны иметь перевод на 4 языка.

## Данные и API
- Все сетевые запросы проходят через существующие hooks/services. Новые эндпоинты документируем в `docs/` + добавляем typing.
- Запрещено хранить чувствительные данные/API-ключи во фронтенде. Используйте серверные прокси/Supabase Edge functions.

## Тестирование и проверка
- Минимум: `pnpm lint`, `pnpm test`, `pnpm build` перед сдачей задачи.
- Для голосового модуля обязательны юнит-тесты hooks + интеграционные тесты API-клиента (см. workflows).
- При фиксе бага добавляйте регрессионный тест, если возможно.

## Логи и аналитика
- Не оставляйте `console.log` в production-коде. Для ошибок используйте централизованные логгеры (Sentry и т.п.).
- Сообщения об ошибках должны быть понятны пользователю и локализованы.

## Документация
- Любые новые возможности документируются в соответствующем разделе `docs/` (например, `docs/ai/voice-assistant/`).
- PR описания должны указывать, какие страницы/языки затронуты, какие тесты прогнаны.

## Проверки перед merge
1. Все тексты локализованы.
2. Все связанные ассеты и meta-файлы обновлены.
3. Линт/тест/bild прошли.
4. Обновлены workflows/документация при необходимости.
5. Нет временных файлов, комментариев и неиспользуемых импортов.
